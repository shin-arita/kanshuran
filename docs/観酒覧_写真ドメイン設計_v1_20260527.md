# 観酒覧 写真ドメイン設計 v1
作成日: 2026-05-27

## 結論

観酒覧MVPでは、
写真は Bottle エンティティに持たせる。

Photo は独立ドメインにしない。

Bottle は最大3枚まで写真を持てる。

ただし、写真は nullable とする。

つまり、

```text
Bottle
└─ photos 0..3
```

で管理する。

---

# 写真の目的

観酒覧における写真の目的は、

- 物理ボトルの識別
- ボトル探索
- 店員の記憶補助

である。

写真はアルバム機能ではない。

---

# ドメインモデル

```text
Bottle 1 ── 0..3 PhotoData
```

ただし PhotoData は独立ドメインではなく、
Bottle の保持情報として扱う。

実装上は、
Bottle に紐づく写真情報として管理する。

---

# 写真枚数

Bottle は最大3枚まで写真を持てる。

想定:

```text
1枚目: ボトル全体
2枚目: ラベル拡大
3枚目: 補助写真
```

補助写真の例:

- 保管状態
- 特徴
- 見分けるための目印

---

# 3枚上限の理由

1枚固定では、

- 暗い
- ラベルが見えない
- 似た瓶と区別しづらい

という問題がある。

一方で無制限にすると、

- アルバム化
- 管理過多
- 入力負荷増加

につながる。

そのため、MVPでは最大3枚とする。

---

# nullable方針

写真は nullable とする。

つまり、

写真0枚でも Bottle 登録可能

とする。

理由:

- 急いで登録する場面がある
- あとで撮影したい場合がある
- 店運用を止めないため

観酒覧は、

```text
まず登録
↓
あとで写真
```

を許容する。

---

# Storage

画像本体はDBに保存しない。

DBでは storage_key を管理する。

初期:

```text
LocalStorage
```

将来:

```text
S3Storage
```

Storage interface を用意し、
LocalStorage から S3Storage へ移行可能な設計にする。

---

# Bottle が持つ写真情報

Bottle は写真情報を持つ。

例:

```text
photo_1_storage_key nullable
photo_2_storage_key nullable
photo_3_storage_key nullable
```

または実装時に、
Bottle配下の写真配列として扱ってもよい。

重要なのは、

- Bottleに紐づく
- 最大3枚
- 画像本体はDB保存しない
- storage_keyを管理する

ことである。

---

# 並び順

写真は表示順を持つ。

基本:

```text
1枚目 = 主写真
2枚目 = 補助写真
3枚目 = 補助写真
```

一覧や検索結果では、
1枚目を代表画像として扱う。

---

# 非採用（MVP OUT）

以下はMVPでは採用しない。

- Photo独立ドメイン
- 写真無制限
- アルバム機能
- 画像本体のDB保存
- OCR
- AI画像認識
- 自動銘柄判定

---

# 設計思想

観酒覧の写真ドメインは、

- 識別
- 探索
- 記憶補助
- 入力負荷を増やさない

を基本とする。

写真は、
ボトル探索を助ける黒子情報である。

観酒覧は、
写真管理システムではない。
