# 観酒覧 ボトル・キープドメイン設計 v1
作成日: 2026-05-27

## 結論

観酒覧MVPでは、
ボトル・キープドメインを Bottle 中心で管理する。

Keep は独立モデルにしない。

キープは Bottle の状態・属性として吸収する。

つまり、

```text
Bottle = キープ対象の物理ボトル兼、預かり状態を持つ管理単位
```

とする。

---

# ドメインモデル

```text
Store 1 ── 0..N Bottle

Bottle 1 ── 0..N BottleLineCustomer
LineCustomer 1 ── 0..N BottleLineCustomer
```

つまり、

```text
Bottle N ── N LineCustomer
```

Bottle は店舗に属する。

Bottle は、LINE連携済みLineCustomerに紐付くこともあれば、
未連携のまま店側だけで管理されることもある。

グループキープ（1ボトル複数LineCustomer）は MVP IN。

---

# Bottle（ボトル）

Bottle は観酒覧の中心管理対象である。

責務:

- 物理ボトルの識別
- 写真管理（最大3枚）
- 銘柄管理
- 保管場所管理
- 残量管理
- 持ち主情報管理
- LINE連携済みLineCustomerとの紐付け（BottleLineCustomer経由）
- 期限管理
- 保管状態管理

観酒覧において店員が探す対象は Bottle である。

---

# Keep の扱い

Keep は独立モデルにしない。

理由:

- MVPでは Bottle と Keep が実質1:1
- 再キープ時は新しい物理ボトルを登録する
- 店員は Bottle と Keep を分けて認識しない
- 店員が見たいのは「どのボトルが、誰のもので、どこにあるか」
- Keep を分けるとMVPとして管理が重くなる

したがって、

```text
Keep = Bottle の状態・属性
```

として扱う。

---

# 再キープ

再キープは既存Bottleの再利用ではなく、
新Bottleの新規作成とする。

例:

```text
Bottle#001
- 森伊蔵
- 残量少ない
- 有田さん
- stored

↓ 飲み切り / 終了

Bottle#001
- finished

↓ 新しいボトルを入れる

Bottle#002
- 森伊蔵
- 残量たくさん
- 有田さん
- stored
```

同じ客の継続関係はLineCustomer側で表現する。

同じBottleを使い回さない。

---

# Bottle が持つ主な項目

```text
brand_name
storage_location_id nullable
remaining_level
owner_name nullable
owner_note nullable
expires_at nullable
status
```

写真は Bottle に最大3枚まで紐づく。Photo独立ドメインは作らない。

---

# 写真

写真は Bottle が持つ。

理由:

写真は「誰のキープか」ではなく、
物理ボトルの識別に使うため。

観酒覧の探索UXでは、

- 写真
- 銘柄
- 保管場所
- 残量

が重要である。

---

# 銘柄

Bottle は銘柄情報を持つ。

例:

```text
森伊蔵
黒霧島
吉四六
```

MVPでは厳密な商品マスタ化を必須にしない。

まずは入力しやすさを優先する。

---

# 保管場所

Bottle は保管場所を持つ。

理由:

保管場所は、

「今どこに物理ボトルがあるか」

を示す情報であり、
探索対象である Bottle の属性だから。

例:

```text
棚A-3
カウンター奥
冷蔵庫右
```

---

# 残量

Bottle は残量を持つ。

残量は status ではなく、
Bottle の属性として扱う。

残量候補:

```text
many
half
low
```

日本語UI例:

```text
たくさん
半分
少ない
```

理由:

現場では正確なml管理より、
ざっくりした残量把握の方が重要なため。

残量は、

- 次回提供準備
- 新ボトル提案
- 店員の記憶補助

に役立つ。

---

# 持ち主情報

LINE未連携時の持ち主情報は、
Bottle 側で持つ。

```text
owner_name nullable
owner_note nullable
```

owner_name は nullable とする。

理由:

現場では、

- 名前未確認
- 団体客
- 仮登録
- メモだけで先に登録

があり得るため。

例:

```text
owner_name = null
owner_note = カウンター右の黒霧
(BottleLineCustomer なし)
```

これは許容する。

---

# LineCustomer連携

LINE連携済みLineCustomerが存在する場合、
BottleLineCustomer を通じて Bottle と LineCustomer が紐付く。

グループキープ時は 1 Bottle に複数 BottleLineCustomer を持てる。

未連携時は BottleLineCustomer なし。

LINE連携後に紐付ける。

---

# 期限

Bottle は期限を持つ。

```text
expires_at nullable
```

期限は nullable とする。

理由:

- 店舗によって期限運用が異なる
- 期限なし運用もあり得る
- MVPでは店運用を止めない

期限切れは自動終了ではない。

期限切れ後も、
終了判断は店が行う。

---

# 状態

Bottle の状態は以下とする。

```text
stored
finished
```

意味:

```text
stored
= 保管中、提供対象

finished
= 終了、履歴対象
```

履歴は独立状態にしない。

finished の Bottle が履歴として扱われる。

---

# 削除

Bottle は原則削除しない。

```text
stored
↓
finished
↓
履歴
```

として扱う。

削除ではなく終了で管理する。

---

# 非採用（MVP OUT）

以下はMVPでは採用しない。

- Keep 独立モデル
- Bottle 1 ── N Keep 履歴モデル
- 同一Bottleの再キープ
- 厳密な残量ml管理
- 厳密な商品マスタ必須化
- 自動期限終了
- Photo独立ドメイン

---

# 設計思想

観酒覧のボトル・キープドメインは、

- 探索対象はBottle
- キープはBottleの状態
- しっかり作るが、管理しすぎない
- 雑でも回る
- 現場入力を止めない

を基本とする。

観酒覧は、

物理資産管理システムではない。

店員が来店客のキープボトルを、
速く、迷わず、記憶依存なく提供するための黒子である。
